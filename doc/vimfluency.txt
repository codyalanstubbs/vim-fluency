*vimfluency.txt*    fluency training sessions for vim drills                       *vimfluency*

==============================================================================
INTRODUCTION                                                       *vf-intro*

Vim Fluency runs short timed training sessions on vim drills to build behavioral fluency.
Each training shows a target buffer state; you produce that state with vim
commands. Per-session rate is logged for progress tracking.

Every drill has an "aim": the rate, in correct items per minute, at
which the behavior counts as fluent. Aims start as rough guesses; set
your own with :VfSetAim.

Two entry points: :Vf opens the dashboard (the home view — browse
drills, launch trainings and lessons from a row with T / L), and
:VfTrain {id} starts a timed session on one drill directly.

Every view links to every other with the same uppercase keys —
T(rain), L(earn), C(hart), I(list), V(dashboard) — so once you've
typed :Vf (or :VfList) you can loop through trainings, lessons, charts,
and the end screen without retyping a command. Q closes / steps back.

==============================================================================
COMMANDS                                                        *vf-commands*

:VfTrain {id} [seconds] [only=motion[,motion...]]               *:VfTrain*
    Start a training on drill {id} for [seconds] (default 60, or
    your :VfSetDuration override). Use Tab to skip an item, :VfQuit
    to end early. When the session ends, the runner lands on the
    shared end screen (see |vf-end-screen|): the drill's last-session
    frequency + per-motion breakdown plus a single-key menu to every
    other view.

    `only=` filters the generator to only produce items where the
    canonical answer is one of the listed motions. Example:
        :VfTrain move_to_line_edges_all only=g_       drill only g_ items
        :VfTrain move_to_line_edges_all only=g_,^     drill the slow pair
    Each generator labels every item with `expected_motion` and the
    end-of-session summary breaks down the rate per motion.

:VfList                                                          *:VfList*
    Open an interactive flat table of installed drills. A column
    header row names the columns; each row then lists, left to right:
        ▶/✓/○         status icon (legend in the banner above the
                      table: ✓ at aim, ▶ climbing, ○ not started).
                      The rule for ✓ is stricter than "the last
                      session was at aim": it requires the LAST
                      THREE non-zero-rate sessions to ALL be at-or-
                      above aim. One good run is performance; three
                      in a row is the Precision Teaching definition
                      of demonstrated stability. So a row can show
                      last_rate well above aim and still display ▶
                      until the trend holds.
        drill         the drill id (what you'd type after :VfTrain)
        commands      the keystrokes that produce this behavior,
                      space-separated
        prereqs_n     transitive prereq depth — 0 = no prereqs,
                      higher values build on lower ones
        aim           fluency aim (rate per minute)
        last_rate     rate of the most recent non-zero session
        last_session  date of that same session
        runs          number of non-zero-rate training runs for this
                      drill (zero-rate quits are excluded so the
                      count matches what last_rate and last_session
                      read from)
        family        the verb family (survival, motion, delete, …)
    Default order: family → prereqs_n → drill, so each family
    clusters and within a family the foundational drills come
    first. There are no per-family section dividers — the family
    column carries the grouping.

    Sort with `s` + a column letter. The active column gets a ▲ (asc)
    or ▼ (desc) marker in the gutter just past the column's right
    edge — header text and marker have a one-col gap between them so
    the marker reads as a separate annotation, not part of the header.
    Pressing the same letter twice flips direction; `s<Space>` clears the sort and
    returns to the default. The tiebreaker is ALWAYS (family, depth,
    slug) ascending — flipping direction only affects the primary key,
    so rows with identical primary values keep the same relative order
    asc-or-desc. After a sort, the cursor stays on the SAME row index
    (the Nth drill from the top), not on the same drill — so
    your eye stays put while the data underneath re-orders. Sort keys:
        sd    drill         (alphabetical by slug)
        sc    commands      (alphabetical by rendered key string)
        sp    prereqs_n     (numeric)
        sa    aim           (numeric)
        sr    last_rate     (numeric; em-dash sorts lowest)
        ss    last_session  (chronological; em-dash sorts lowest)
        sn    runs          (numeric)
        sf    family        (curated FAMILY_NAMES order)

    The banner, status legend, sort hints, and column-header row live
    in a small STICKY-HEADER window above the scrollable data window.
    The header stays visible while you scroll through the rows; sort
    markers update there when you change the sort. The cursor lives
    in the data window. Either `Q` or `:q` closes both windows and the
    tab in one shot (lowercase `q` still works as an alias). The
    cursor row is highlighted; j/k snap between
    drill rows (breakdown rows are skipped). On a drill row
    press:
        L     start a Learn session (:VfLearn) for this drill
        T     start a Train session (:VfTrain) for this drill
        C     open the progress Chart (:VfChart) for this drill
        V     open the dashboard (:Vf) on this drill
        B     toggle the Breakdown beneath the drill — see below
        A     prompt to set the aim for THIS drill. Type a
              positive integer to store an override (same effect as
              :VfSetAim), 0 to clear an existing override (same as
              :VfResetAim), or press Esc to cancel.
        D     prompt to set the GLOBAL default :VfTrain duration in
              seconds. Same semantics as A: positive int stores
              (:VfSetDuration), 0 clears (:VfResetDuration), Esc
              cancels.
        Q     close the list (lowercase q also works)
    gg / G jump to the first / last drill. C requires logged
    sessions, L requires a lesson, B requires either a multi-command
    session or at least one prereq — each is a no-op with a hint
    otherwise. Sort state is per-buffer and does NOT persist — a
    fresh :VfList starts at the default.

    The B breakdown shows up to two sub-blocks beneath the drill:
        prereqs:   every in-registry prereq, with its own ▶/✓/○ icon
                   so you can see at a glance which are still
                   climbing
        commands:  a sub-table per command from the last session —
                   command, last_rate, stroke_count (physical
                   keystrokes including modifiers like shift), and
                   stroke_rate = last_rate / stroke_count (a fairness
                   lens for comparing 1-key and multi-key commands)
    stroke_count is derived from the command string (shift counts as
    its own keystroke, and Ex commands add one for the trailing
    <Enter>, so `q!` = 3 and `:wq` = 5); a drill can declare
    `stroke_counts: {motion → N}` in meta() to override the derived
    count for a specific command.

:Vf [id]                                                             *:Vf*
    Open the multi-panel dashboard: a one-line banner (current path,
    status counts, sessions trained), a hover panel showing progress
    charts for the drill under the cursor, the interactive drill
    table, and a LAST SESSION side panel with the hovered drill's most
    recent breakdown (date / rate / aim / efficiency, prereqs,
    per-command sub-table). The table rows and j/k navigation work
    like :VfList; the hover and LAST SESSION panels refresh as the
    cursor moves. The table is filtered to the current path (see
    :VfSetPath); :VfTrain {id} and :VfLearn {id} still accept every
    drill regardless of path. Optional [id] lands the cursor on
    that row (the shared end screen's `d` key opens the dashboard
    here, on the just-finished drill). If a dashboard tab is already
    open it is reused and rebuilt in place rather than duplicated.
    Keys on the
    table window:
        L T C A D     same as :VfList (Learn, Train, Chart, Aim,
                      Duration)
        I             open the flat :VfList view
        P             prompt to switch the learning path (Tab
                      completes; 'general' or :VfResetPath reverts)
        B             jump into the LAST SESSION panel to scroll a
                      breakdown that overflows the window
        Q             close the dashboard (lowercase q also works)
    Inside the LAST SESSION panel: j/k scroll, J or <CR> on a prereq
    line jumps the table cursor to that prereq's row, and Q, <Esc>,
    or B return to the table (lowercase q also works).

:VfQuit                                                          *:VfQuit*
    End the active session, write its log entry, restore the window layout.

THE END SCREEN                                              *vf-end-screen*
    Both :VfTrain and :VfLearn finish on the same screen. It shows the
    drill's most recent training session's frequency (corrects per
    minute) and per-command breakdown — the same LAST SESSION panel the
    dashboard draws. A drill you've only ever done a lesson on has no
    rate yet, so the panel shows an empty-state nudge to run a training
    and log your first data point. From the screen, single keys jump to
    any other view (keys are uppercase, matching the dashboard /
    :VfList convention):
        T   train this drill        (:VfTrain {id})
        L   learn this drill        (:VfLearn {id})
        C   chart your progress     (:VfChart {id})
        I   browse all drills       (:VfList)
        V   open the dashboard      (:Vf {id})
        Q   quit

:VfHistory [id]                                                *:VfHistory*
    Show prior sessions from the JSONL log, grouped by drill and
    rendered with an ASCII rate bar (filled = fraction of aim, all '*'
    means at-or-above aim). Optional [id] filters to a single drill.
    For groups with n≥2 sessions, shows the first→last rate ratio.

:VfLearn {id}                                                    *:VfLearn*
    Walk through a scripted lesson for drill {id} (DI-style example/
    non-example sequencing followed by 'try it' frames) before timing the
    skill with :VfTrain. Press <Space> or <CR> to advance through 'show'
    frames; reach the green target to advance through 'try' frames; press
    'Q' or :VfQuit to stop early (lowercase q also works). Drills
    without a lesson() function
    will report "no lesson written for {id} yet". On completing the
    lesson (the required first-try streak in the test phase) you land
    on the shared end screen (see |vf-end-screen|) — the same screen
    :VfTrain ends on.

:VfChart {id}                                                    *:VfChart*
    Render a progress chart for drill {id} — a Standard Celeration
    Chart, the Precision Teaching convention for plotting rates on a
    semi-log scale. Y-axis is rate per minute on a log scale (1 to 1000 across
    three decades). X-axis is session index (chronological). Plots both
    the corrects rate (●) and the errors rate (×) per session, with the
    fluency aim drawn as a dashed line. Navigation keys (you never have
    to retype a command to move on):
        T   train this drill        (:VfTrain {id})
        L   learn this drill        (:VfLearn {id})
        Z   toggle the zoom variant (:VfChart / :VfChartZoom)
        I   browse all drills       (:VfList)
        V   open the dashboard      (:Vf {id})
        Q   close (or <Enter>; lowercase q also works)

:VfChartZoom {id}                                            *:VfChartZoom*
    Same as :VfChart but with the Y-axis zoomed to a single decade (10
    to 100). Useful when all sessions sit in the middle of the full
    1-1000 range and the full chart wastes most of its vertical space.
    Sessions or aim lines outside 10-100 are clipped.

:VfSetAim {id} {rate}                                          *:VfSetAim*
    Store a personal aim override for drill {id} (rate per minute).
        :VfSetAim insert_before_after_char_start_end_line 75
    The override persists across vim restarts in
    $XDG_DATA_HOME/vimfluency/settings.json and is used everywhere the
    aim matters: :VfList rows show the effective value (with a trailing
    `*` on overridden rows), the at-aim status calculation reads the
    override, :VfChart draws the aim line at the override, and the
    breakdown's ✓ mark on per-command rates uses it. The default in
    meta() stays untouched.

:VfResetAim {id}                                            *:VfResetAim*
    Clear the personal aim override for drill {id}, reverting to
    the drill's meta.aim default.

:VfSetDuration {seconds}                                  *:VfSetDuration*
    Store a personal global default duration (seconds) for :VfTrain runs
    that don't specify an explicit duration.
        :VfSetDuration 90
    Per-invocation arg on :VfTrain <id> <seconds> still wins over both this
    override and the 60s built-in default. There is no per-drill
    duration override; this is one knob that applies to every :VfTrain
    that doesn't pass an explicit duration.

:VfResetDuration                                        *:VfResetDuration*
    Clear the personal default duration override (revert to 60s).

:VfSetPath {name}                                            *:VfSetPath*
    Store the current learning path. A path is a curated subset of
    the drill registry (defined by a file under
    autoload/vimfluency/paths/) that filters what :Vf shows.
        :VfSetPath foundational
    Tab-completion offers the discovered paths; unknown names are
    rejected with the list of what's available. The selection
    persists in settings.json as `current_path`. Built-in paths:
        general        every shipped drill, no curation (default)
        foundational   survival skills + basic motion — mode
                       switching, positional inserts, save/quit,
                       undo, hjkl, line edges, word motion
    Path ids listed in a path file but not (or no longer) in the
    registry are silently dropped, so paths survive drill renames.

:VfResetPath                                               *:VfResetPath*
    Clear the path selection, reverting to 'general'.

:VfPaths                                                       *:VfPaths*
    List the available paths, one per line: a ▶ marker on the current
    selection, the path id, its display name, and how many shipped
    drills it covers.

==============================================================================
LOG AND SETTINGS                                                     *vf-log*

Each session appends one JSON line to:
    $XDG_DATA_HOME/vimfluency/sessions.jsonl   (if XDG_DATA_HOME is set)
    ~/.local/share/vimfluency/sessions.jsonl   (otherwise)

Fields per record: timestamp, drill_id, drill_name, aim,
duration_seconds, elapsed_seconds, items_correct, items_skipped,
items_hit, frequency_per_min, hits_per_min, miss_per_min,
errors_per_min, total_motions, total_optimal_motions, efficiency_pct,
end_reason, only_filter, per_motion, items. The recorded `aim` is the
EFFECTIVE aim at training time (the user override if one is set,
otherwise the drill's meta.aim default). items_hit counts correct
items solved with no wasted motions (actual ≤ optimal); hits_per_min /
miss_per_min are the Standard-Celeration-Chart-convention rates (see
:VfChart) derived from hits, wasted
motions, and skips. errors_per_min is wasted motions (total minus
optimal, clamped at 0) per minute. per_motion maps each
expected_motion to {correct, time_seconds, rate_per_min, avg_motions,
avg_optimal}; items is the full per-item log.

User settings live alongside the log in:
    $XDG_DATA_HOME/vimfluency/settings.json

with the shape:
    {
      "aims":             {<drill_id>: <integer rate per minute>},
      "default_duration": <integer seconds>,
      "current_path":     <path id>
    }

Set and clear via :VfSetAim, :VfSetDuration, and :VfSetPath; the file
is rewritten on each change. A missing or malformed file is treated as
"no overrides" so neither :VfList nor :VfTrain ever errors on a fresh
install.

Renamed drills: when a drill's id changes between releases, the old id
keeps working — session records in the log, aim-override keys in
settings.json, and ids typed into :VfTrain / :VfLearn / :VfChart /
:VfHistory are all mapped to the current id at read time. The log
file itself is never rewritten, so history and charts stay continuous
across a rename.

ADDING A DRILL                                                 *vf-drills*

A drill targets one precisely specified behavior — what Precision
Teaching calls a "pinpoint". This project used that term internally
before standardizing on "drill"; it now survives only in the legacy
pinpoint_id / pinpoint_name fields of old session-log lines, which are
still read for back-compat.

Drop a file at  autoload/vimfluency/drills/<slug>.vim  defining:

    function! vimfluency#drills#<slug>#meta() abort
        return {'id': '...', 'name': '...', 'aim': N, 'allowed_keys': '...'}
    endfunction

    function! vimfluency#drills#<slug>#generate() abort
        return {'lines': [...], 'start': [row, col], 'target': [row, col]}
    endfunction

{slug} is a descriptive snake_case identifier starting with a letter
(e.g. move_single_char_left_right) — it is both the filename minus
.vim and the {id} users type after :VfTrain.
Rows and cols are 1-indexed (vim convention).
'allowed_keys' is advisory documentation of the intended key set; the
runner does not restrict input with it.

TRAINING KINDS                                                    *vf-kinds*

meta() may declare a `kind` field:

    motion (default)
        Cursor-only training. Item declares `lines`, `start`, `target`.
        Runner credits when the cursor reaches `target` without
        modifying the buffer.

    editing
        Buffer-changing training (operators: x, dd, dw, ...). Item
        adds `target_lines` (post-edit buffer state) and may add
        `deletion_range` for the red highlight bar. Runner credits
        when the buffer matches `target_lines` AND cursor lands at
        `target`.

    recall
        Type-the-answer training (e.g., name a mode, type a save
        command). Item declares `prompt` (string or list of lines)
        and `expected_answer` (string). The runner auto-credits the
        moment the typed input exactly equals `expected_answer` —
        no <CR> required. <BS> corrects; <Tab> skips.

    mode
        Round-trip-through-insert training (e.g., i / a / o / O). Item
        adds `enter_at_row`, `enter_at_col` (where InsertEnter must
        fire — this is what distinguishes i from a, I from A by the
        cursor position when insert begins), `target_lines`
        (post-Esc buffer state), and `target` (post-Esc cursor pos
        in normal mode). Runner credits on InsertLeave when all
        three match.

    mode_switch
        Mode-change production training (the switch_mode_* family).
        Item declares `target_mode_canon` — one of n, i, v, r, c
        (visual char/line/block collapse to v; Replace variants to
        r; command-line variants to c). Runner credits the moment
        vim's canonical mode matches the target (via ModeChanged on
        vim 8.2+, or 50ms mode() polling on 8.1). The next item's
        target is generated with a no-repeat constraint (target !=
        current mode).

    command
        Ex / normal-mode command training (e.g., :wq vs ZZ). Item
        declares a code `snippet` plus a `goal` rendered as a
        comment line above it; the buffer is nomodifiable. Buffer-
        local mappings capture ':' (a fake cmdline via input()) and
        the ZZ / ZQ shortcuts, so the command is checked but never
        executed — vim doesn't quit, the snippet isn't written.
        Credits when the typed command equals `expected_motion`;
        wrong commands echo a hint and the learner retries — mistakes
        are never auto-failed. :VfQuit typed at the fake cmdline still
        passes through and ends the session.

    visual_motion
        Charwise visual-selection training (the visual_select_*
        family, e.g., vh / vj / vk / vl). Item adds
        `expected_selection_start`, `expected_selection_end`, and
        `expected_sub_mode` (default 'v'; sub-modes are deliberately
        NOT collapsed — v-family drills discriminate them). Runner
        credits when mode(1) equals the expected sub-mode AND the
        selection anchor (getpos('v')) and cursor match the expected
        range, counting the v keystroke toward the motion total,
        then drops the learner back to normal mode so the next item
        starts clean.

REQUIREMENTS                                                       *vf-reqs*

Vim 8.1 or later (for rand() — patch 8.1.2342 — json_encode(),
timer_start()), or Neovim. The legacy-vimscript baseline runs on
Neovim unmodified; verified on nvim 0.11 (headless test suite,
scripted interactive smoke test, and real interactive use).
